# kitchen-sink — Ream Quick Start

A working demo that exercises every framework package end-to-end so you can read it like a tutorial. It's not a "hello world" — it's the **shape of a real app** boiled down to ~2k lines of TypeScript.

```bash
pnpm install
pnpm -w build   # builds aurora's browser dist (one-time, ~1s)
pnpm test       # 73 e2e + smoke tests, helix runner, --threads=1
pnpm dev        # http://localhost:3000
```

In the workspace the apps run the packages' **TypeScript source directly** — every package's `exports` point at `src/*.ts`, so editing `packages/*/src` is picked up immediately with no rebuild (the dev loop runs through `@swc-node/register` / `tsx`). No `--conditions` flag is needed.

`pnpm -w build` compiles every package's `src/*.ts` into `dist/*.js` (+ `.d.ts`). That compiled output is **only** consumed at publish time: each package's `publishConfig` repoints `main`/`exports` at `dist/` so npm consumers without a TS transform get plain JS, while the checked-in `package.json` stays source-first. `dist/` is git-ignored — it's a build artifact, regenerated by `pnpm build` before `pnpm publish`.

## App layout

```
app/
  middleware/
    auth_middleware.ts      # populates ctx.auth from Bearer; opt-in via .guard('jwt')
  modules/
    workspace/              # workspaces + memberships + invitations
    project/                # projects scoped to a workspace, public/private visibility
    task/                   # tasks, recurrence, status transitions
    comment/                # task comments (XSS-stripped via blackhole)
    attachment/             # archive-backed blobs, base64 wire format
    notification/           # events → bay → mail/push fan-out
    site/                   # aurora SSR + hydrate demo (public)
    auth/                   # sign-up / sign-in
config/                     # one file per framework package
database/                   # migrations
resources/lang/             # rosetta translation bundles
resources/pages/            # shared aurora pages (SSR + hydrate, plain ESM)
start/
  kernel.ts                 # server middleware + named middleware
  services.ts               # event subscriptions + relay authorizer
bin/server.ts               # Ignitor entrypoint
reamrc.ts                   # provider list, preloads, module autoload
tests/
  smoke/                    # surface checks (package exports)
  e2e/                      # full HTTP roundtrips through HyperServer
```

A module is a folder with `routes.ts` (auto-loaded) + a `controllers/` directory. Adding `events.ts` registers event listeners. Nothing else is required.

## Auth flow

The global auth middleware reads `Authorization: Bearer <token>` and stores `ctx.auth.user` if the token verifies. It runs on every request but **never rejects**:

```ts
// start/kernel.ts
server.use([
  () => import('@c9up/blackhole/middleware'),
  () => import('#middleware/auth_middleware.js'),
])
```

Gating happens at the route level with `.guard('jwt')`:

```ts
// app/modules/workspace/routes.ts
router.post('/workspaces', [WorkspaceController, 'create']).guard('jwt')
router.get('/workspaces', [WorkspaceController, 'list']).guard('jwt')
```

Inside a guarded controller you can dereference `auth.user!.id` without any boilerplate — the guard already rejected anonymous requests with 401:

```ts
async create({ request, response, auth }: HttpContext) {
  const callerId = auth.user!.id as string
  // …
}
```

Public routes simply omit `.guard('jwt')`. They can still read `auth.user?.id` defensively — see `ProjectController.show` which serves public projects to anonymous viewers.

Project-membership access lives in a single helper:

```ts
// ProjectService
async canAccess(projectId: string, userId: string): Promise<boolean> {
  const project = await this.projects.find(projectId)
  if (!project) return false
  const membership = await this.workspaces.getMembership(project.workspaceId, userId)
  return membership !== null
}
```

Task, Comment and Attachment controllers call `projectSvc.canAccess(task.projectId, callerId)` and 403 on `false`.

## Realtime — relay (SSE)

The relay package auto-registers three routes during `start`:

- `GET  /__relay/events`        — long-lived SSE stream
- `POST /__relay/subscribe`     — register a channel against the client's stream
- `POST /__relay/unsubscribe`   — drop a subscription

Customise them in your app's preloads (e.g. add the `jwt` guard so anonymous SSE is refused):

```ts
// start/services.ts
relay.registerRoutes((route) => route.guard('jwt'))
```

Authorize a channel and broadcast on a domain event:

```ts
relay.authorize<{ id: string }>('project/:id', async (ctx, { id }) => {
  const userId = ctx.auth?.user?.id
  if (typeof userId !== 'string') return false
  const project = await projectSvc.projects.find(id)
  if (!project) return false
  const membership = await wsSvc.getMembership(project.workspaceId, userId)
  return membership !== null
})

emitter.on(TaskAssigned, async (e) => {
  relay.broadcast(`project/${e.projectId}`, {
    event: 'task.assigned',
    taskId: e.taskId,
    assigneeId: e.assigneeId,
    byUserId: e.byUserId,
  })
})
```

Browser client:

```js
const sse = new EventSource('/__relay/events', { withCredentials: true })
sse.onmessage = (e) => console.log(JSON.parse(e.data))

await fetch('/__relay/subscribe', {
  method: 'POST',
  headers: { 'content-type': 'application/json' },
  body: JSON.stringify({ uid: sse.uid, channel: `project/${projectId}` }),
})
```

`sse.uid` is the stream id the server sends on the connection's first frame.

## Background work — events + bay

Domain events emit immediately. Heavy fan-out (DB writes, mail, push) runs on bay so HTTP handlers return fast:

```ts
// start/services.ts
queue.register(
  'notify-assignee',
  app.container.make<NotifyAssigneeJob>(NotifyAssigneeJob), // ← container-resolved
)

emitter.on(TaskAssigned, async (e) => {
  await queue.dispatch('notify-assignee', { taskId: e.taskId, /* … */ })
})
```

**Important:** bay calls `new Handler()` with no args. Any handler with `@inject()` deps must be resolved through the IoC container *before* `queue.register` — otherwise the handler runs with `undefined` services.

Tests drain the queue inline (`queue.processOne()`) from `NotificationController.list`. In production a separate worker process consumes the queue.

## i18n — rosetta

```ts
// app/modules/project/controllers/ProjectController.ts
const locale: 'fr' | 'en' = localeRaw === 'en' ? 'en' : 'fr'
response.json({
  ok: true,
  project: shape(project),
  description: svc.descriptionForLocale(project, locale),
  locale,
})
```

`?locale=fr|en` query string overrides Accept-Language. The site SSR demo (photon + aurora) goes further and uses the rosetta loader to render template strings server-side.

## SSR + hydrate — aurora (Inertia-shape)

```ts
// app/modules/site/controllers/SiteController.ts
import aurora from "@c9up/aurora/services/main"

await aurora.render(ctx, "ProjectPage", props, { lang, headExtra })
```

`ProjectPage` lives in `resources/pages/ProjectPage.js` as **plain ESM** — same source on the server (loaded by `@swc-node/register`) and the browser (loaded through the importmap that the helper auto-injects). The page wraps its setup in `component()`, holds reactive state in `signal()`, and subscribes to relay channels via `onMount(() => relay().subscribe(...))`. Aurora's provider auto-mounts `GET /_assets/aurora/*` and `GET /_assets/pages/*` — no asset controller in the app.

## Patterns to copy

1. **Routes file = auth gate.** `.guard('jwt')` opts in; public routes stay bare.
2. **Auth middleware never rejects.** Its job is to populate `ctx.auth`; the guard is what enforces.
3. **Service helpers, not per-controller booleans.** `ProjectService.canAccess` lives once.
4. **`services/main` singletons everywhere.** `import queue from '@c9up/bay/services/main'` — no managers spun up by hand.
5. **Container-resolve job handlers.** `app.container.make(Handler)` before `queue.register`.
6. **Customise framework routes via `registerRoutes`.** Relay's three SSE routes get the same `.guard('jwt')` treatment as everything else.
